# Домашнее задание к занятию "`Базы данных, их типы`" - `Бурыкин Кирилл`

##Задание 1. СУБД
###1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.
Лучше использовать реляционную СУБД (PostgreSQL, MySQL, Oracle) данные не потеряются,все четко по таблицам,возможность строить 
###1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?
Можно использовать In-memory базы данных (например, Redis) как кэширующий слой, чтобы снизить нагрузку на основную СУБД и ускорить операции хеширования.
###1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.
Лендинги: Гибкая база без строгих таблиц (MongoDB) — можно легко менять структуру
CRM: База с таблицами (PostgreSQL) или тоже гибкая, если нужно
###1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?
Да, можно взять PostgreSQL — она поддерживает работу с таблицами и гибкие JSON-документы.
###1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.
Простая база с таблицами (PostgreSQL или даже SQLite). Будет понятно как в паках на ПК.
###1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?
Да, можно сделать в той же базе, что и для бюджетирования, но в отдельном разделе.
###1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.
Графовая СУБД (Neo4j), так как она эффективно работает со связями (маршруты, точки доставки, связи между объектами).
###1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?
Лучше отдельную табличную базу, но связать с логистикой.
###1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?
PostgreSQL — она многое умеет. Но для больших проектов лучше разные базы для разных задач.

##Задание 2. Транзакции
###2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.
1. Начать операцию
2. Проверить номер и деньги
3. Снять деньги с карты
4. Зачислить на телефон
5. Подтвердить, если всё ок
6. Записать что сделали
###2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?
Дополнительные шаги:
· Проверка подписки на автоплатёж.
· Автоматическое инициирование операции по расписанию.
· Уведомление пользователя о списании.

##Задание 3. SQL vs NoSQL
###3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.
1. Надёжнее — данные не потеряются
2. Единый язык SQL везде одинаков
3. Связи между таблицами работают чётко
4. Проверенные технологии — работают давно
5. Консистентность — все видят одни данные
###3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.
· Быстрая как NoSQL
· Надёжная как SQL
· Масштабируется легко

##Задание 4. Кластеры
Для большого количества вычислений на 1000 машинах лучше выбрать колоночную СУБД (ClickHouse, Amazon Redshift) или NewSQL (CockroachDB).
Критерии выбора:

· Масштабируемость.
· Поддержка распределённых вычислений.
· Скорость обработки аналитических запросов.

Модель распределённых вычислений: MPP (Massively Parallel Processing) — данные распределены по узлам, каждый узел обрабатывает свою часть, что ускоряет вычисления.
